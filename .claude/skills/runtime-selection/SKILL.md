# Runtime Selection — 크롤링/추출 서비스 런타임 결정

URL 메타데이터(Open Graph) 추출 서비스의 런타임을 **작업 특성 × 생태계 × 조직 제약**의 함수로 결정하는 절차. 결론을 먼저 내되 결정 매트릭스로 방어한다.

## 언제 이 스킬을 쓰는가
"Node냐 Python이냐", "어떤 언어/프레임워크로 크롤러를 만들까", 기술 스택 선정. 추출/크롤링/스크래핑 워크로드의 런타임 결정에 특화.

## 판단 절차

1. **워크로드 특성 규정** — 이 작업이 무엇에 바운드되는지 확정한다.
   - OG 추출은 **압도적으로 I/O 바운드**다 (HTTP 페치 대기가 지배적). CPU는 HTML 파싱 정도로 가볍고, 무거운 렌더링은 헤드리스 브라우저(별도 프로세스)로 오프로드된다.
   - 따라서 "많은 동시 연결을 낮은 오버헤드로" 다루는 능력이 런타임의 핵심 요건이다.

2. **평가 축에 가중치 부여** — 조직 맥락에 따라 가중치를 조정한다.

3. **결정 매트릭스 채점** — 아래 표를 프로젝트 맥락으로 재채점한다.

4. **반대 선택 조건 확인** — 기본 권고를 뒤집는 조건이 있는지 점검.

5. **격리 설계 권고** — 크롤러를 별도 서비스로 분리해 결정을 되돌릴 수 있게 한다.

## 결정 매트릭스 (그린필드 기준선, 1~5)

| 평가 축 | 가중치 | Node.js (TS) | Python | 근거 |
|---|---|---|---|---|
| OG/크롤 라이브러리 생태계 | ★★★ | 5 | 4 | Node: metascraper(모듈형 룰/폴백 최강), open-graph-scraper, unfurl, cheerio. Python: BeautifulSoup, extruct(microdata/RDFa/JSON-LD), metadata_parser |
| 헤드리스 브라우저 지원 | ★★★ | 5 | 4 | Playwright는 양쪽 다 1급. Puppeteer는 Node 전용. Node가 예제/생태계 미세 우위 |
| 동시성 모델(I/O 바운드) | ★★★ | 5 | 4 | Node 이벤트 루프는 다수 동시 페치에 자연스러움. Python은 asyncio(httpx/aiohttp)로 대등하나 sync+thread는 무거움 |
| 팀 역량/기존 스택 | ★★★★ | 맥락 | 맥락 | **대개 결정적 축.** 주변 서비스 언어에 맞추면 운영 표면이 준다 |
| 배포·관측 SDK | ★★ | 4 | 4 | OTel/Prometheus/Datadog 모두 양쪽 지원. 조직 표준을 따른다 |
| 데이터/ML 후처리 | ★★ | 3 | 5 | 콘텐츠 분류·임베딩·언어감지 등 후처리가 크면 Python 우위 |

> 가중치는 예시다. **팀 역량/기존 스택 축이 대개 다른 모든 축을 압도**한다 — 이 점을 결론에 반드시 반영한다.

## 기본 권고

**그린필드 신규 마이크로서비스면 Node.js(TypeScript)를 기본값으로 한다.** 근거:
- metascraper/open-graph-scraper 생태계가 OG 추출 도메인에 가장 성숙 (룰 기반 폴백, oEmbed/JSON-LD/Twitter Card 통합).
- Playwright 1급 + Puppeteer로 헤드리스 옵션이 넓다.
- 이벤트 루프가 I/O 바운드 크롤링에 구조적으로 잘 맞아 적은 코드로 높은 동시성.
- 프론트엔드(SSR/미리보기 렌더)와 언어 공유 시 이점.

## 반대 선택 조건 (Python을 택하라)

- **조직 플랫폼이 Python 중심**이다 (Django/FastAPI 기반, Python 관측/배포/CI 표준) → 운영 일관성이 라이브러리 미세 우위를 이긴다.
- 팀이 Python에 확연히 강하다 (유지보수·채용).
- 추출 후 **무거운 데이터/ML 후처리**(콘텐츠 분류, 임베딩, 요약)가 로드맵에 있다.
- 사내에 재사용할 크롤링/파이프라인 자산이 Python으로 존재한다.

## 격리 설계 권고 (결정을 되돌릴 수 있게)

런타임 선택의 리스크를 낮추는 가장 좋은 방법은 **크롤러를 언어 결정이 갇히지 않는 경계로 격리**하는 것이다:
- 크롤/추출을 **독립 마이크로서비스**로 두고 나머지 서비스와 HTTP/gRPC/큐로만 통신 → 언어 교체 시 이 서비스만 재작성.
- 헤드리스 렌더는 **별도 워커 풀**로 더 격리 (리소스 프로파일이 다름). 이 계층은 언어 무관하게 교체 가능.
- 계약(요청 URL → OG payload JSON)을 언어 중립 스키마로 고정.

## 출력 형식

```markdown
# 런타임 선정: {권고 한 줄}

## 권고
{Node/Python} — {핵심 근거 2~3개}

## 결정 매트릭스
{프로젝트 맥락으로 재채점한 표}

## 언어별 라이브러리 맵
- 페치: {node: undici/got ; python: httpx}
- 파싱: {node: cheerio/metascraper ; python: selectolax/BeautifulSoup + extruct}
- 헤드리스: {Playwright / Puppeteer}

## 반대 선택 조건
{이 프로젝트에서 Python을 택해야 하는 조건 점검 결과}

## 격리 설계
{크롤러 마이크로서비스 분리 + 헤드리스 워커 분리 권고}
```

## 원칙
- **취향이 아니라 함수로 결정한다.** 매트릭스 없이 "Node가 좋다"라고 쓰지 않는다.
- **기본 권고 + 반대 선택 조건을 항상 병기**한다. 조직 맥락 정보가 없으면 그린필드/기존스택 두 시나리오로 분기한다.
- 언어 결정보다 **격리 설계**가 리스크를 더 줄인다는 점을 강조한다.
