# Design Integration Review — 경계면 정합성 교차 검증

개별 산출물의 완성도가 아니라 **산출물들 사이의 경계면**을 검증한다. 각자 옳아도 경계에서 어긋나면 통합 버그가 된다. 핵심: **존재 확인이 아니라 두 산출물을 동시에 열고 필드/계약을 나란히 대조**.

## 언제 이 스킬을 쓰는가
여러 설계 산출물 + 구현의 정합성 검증, 통합 리뷰, QA. 각 산출물 완성 직후 점진적으로(incremental) 수행.

## 검증할 경계면 (OG 추출 하네스)

| # | 경계면 | 대조할 두 필드 | MISMATCH 예시 |
|---|---|---|---|
| 1 | 페치 스키마 ↔ 캐시 key | crawl-engine의 `normalized_url`/`final_url`/`canonical_url` ↔ platform의 캐시 key 입력 | 엔진은 canonical을 안 내는데 아키텍처는 canonical로 캐시 |
| 2 | 에러 taxonomy (단일 진실) | crawl-engine 에러 코드 집합 ↔ reliability-ops 메트릭 라벨/재시도 분류 | 엔진은 `HTTP_429`, 운영은 `RATE_LIMITED`로 라벨 → 집계 깨짐 |
| 3 | 저장소 요구 ↔ DB 선택 | reliability-ops의 SQL 집계 요구 ↔ platform의 DB 선택 | 운영은 도메인 집계 필요한데 아키텍처가 Redis-only 제안 |
| 4 | 런타임 전제 ↔ 엔진 요구 | crawl-engine이 가정한 라이브러리(헤드리스 등) ↔ runtime 선택 | 엔진이 Puppeteer 가정, 런타임은 Python 선택 |
| 5 | per-domain 규칙 계약 | reliability-ops 규칙 필드 ↔ crawl-engine이 소비하는 필드 | 운영이 `wait_selector` 정의, 엔진이 소비 안 함 |
| 6 | 구현 ↔ 설계 | implementation 코드의 캐시key/SSRF가드/에러코드 ↔ 설계 결정 | 설계는 홉마다 SSRF 검증, 코드는 최초 1회만 |

## 검증 절차

1. **경계면별로 두 산출물을 동시에 연다.** 한 문서만 읽고 판단하지 않는다.
2. 각 경계면의 두 필드를 나란히 놓고 **정확히 같은 이름/타입/의미인지** 대조.
3. 검증 가능한 항목(캐시 key 정규화 결과, SSRF 차단 등)은 **실제 스크립트로 확인**.
4. 판정: **PASS / MISMATCH / GAP**.
   - MISMATCH: 두 산출물이 충돌 → 어느 필드가 어떻게 다른지 지목 + 수정 지시.
   - GAP: 한쪽이 비어 검증 불가 → 무엇이 없는지 명시.
5. MISMATCH/GAP는 해당 에이전트에게 재작업 요청. **삭제·임의 수정 금지**, 출처 병기.
6. 각 산출물 완성 직후 **점진 검증** — 전체 완성 후 1회가 아니라.

## 출력 형식
```markdown
## 경계면 검증 결과
| 경계면 | 산출물 A 필드 | 산출물 B 필드 | 판정 | 근거 | 수정 지시 |
|---|---|---|---|---|---|
| 에러 taxonomy | crawl-engine: HTTP_429 | ops: RATE_LIMITED | MISMATCH | 라벨 불일치로 집계 불가 | ops를 HTTP_429로 통일 |

## 종합
- PASS: N / MISMATCH: M / GAP: K
- 잔여 이슈 및 재작업 요청 대상
```

## 원칙
- **경계면 교차 비교가 핵심.** "각 문서가 그럴듯하다"는 검증이 아니다.
- **점진 검증**: 각 산출물 완성 직후. 통합 버그는 이르게 잡을수록 싸다.
- **모순은 병기·보고**하고 재작업 요청. 강제 통과하지 않는다(2회 재작업 후에도 남으면 상위 보고).
